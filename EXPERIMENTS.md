# 파인튜닝 실험 가이드 — 세팅 · 실행 · 로드맵

> 최종 갱신: 2026-07-13 | 전제 문서: `HANDOVER.md`(경위), `outputs/zero_shot_report.md`(zero-shot 결론)
> 이 문서는 "실험을 어떻게 세팅했고, 앞으로 어떤 순서로 무엇을 실험하는가"를 다룬다.

## 1. 실험 원칙 (모든 실험에 적용)

1. **같은 자**: 평가는 항상 `splits/holdout_300.csv` 고정. 다른 셋으로 잰 숫자는 비교표에 올리지 않는다.
2. **같은 채점**: Exact Match, 핵심 지표는 **`acc_shuffled`(섞인 샘플 정확도)**.
   - 무작위 기준선 = 1/24 ≈ **4.2%**, zero-shot 전 모델 0.8~4.4% (= 출발선 0)
   - 전체 정확도(`accuracy`)는 identity 응답 빈도에 오염되므로 참고용
3. **같은 장부**: 모든 결과는 `outputs/experiments.csv`에 자동 누적. 수기 기록 금지.
4. **변수는 한 번에 하나**: 두 개 이상 바꾸면 어떤 게 효과였는지 알 수 없다.
5. **노이즈 판정**: holdout의 섞인 샘플은 252개 → **acc_shuffled 차이가 ±4%p 이내면 동급**으로 취급 (이미지 EDA의 McNemar 교훈).
6. **학습·평가 프롬프트는 세트**: 프롬프트를 바꾸면 학습 데이터 생성과 평가 양쪽을 같이 바꿔야 한다. 평가만 바꾸는 실험은 무효.

## 2. 인프라 구성 (2026-07-13 구축, 스모크 검증 완료)

| 구성 요소 | 역할 |
|---|---|
| `scripts/train.py` | 학습 엔진. 재셔플 증강, 검증셋 Id 자동 제외, LoRA bf16, 밤샘 안전장치 |
| `scripts/eval_zero_shot.py --adapter` | 파인튜닝 어댑터 평가. `acc_shuffled`/`acc_identity`/`prompt` 자동 기록 |
| `scripts/prompts.py` | 프롬프트 레지스트리 — train/eval이 `--prompt <이름>`으로 공유 |
| `Train_Experiments.ipynb` | 조종석: 실험 레지스트리(설정 딕셔너리) → 스모크→학습→평가→비교표→오답 분석 |
| `outputs/runs/<run-name>/` | 실험별 산출물: `adapter/`, `train_log.csv`, `run_config.json` |
| `outputs/experiments.csv` | 전 실험 비교 장부 (`model_id` = `<모델경로>+<run-name>`) |
| `outputs/errors_<모델>_<run>.csv` | 실험별 오답 상세 (텍스트 EDA 담당 공유용) |

**자동으로 지켜지는 것들** (train.py 내장이라 신경 안 써도 됨):
- 학습 제외 Id = holdout_300 ∪ stratified_valid = **494개** (평가 오염 방지)
- 재셔플 증강: 샘플당 `--aug-mult`개 제시 순서 변형(변형 0 = 원본), 시간순 라벨 자동 재계산
- 프롬프트 = eval과 동일 문자열 (개선분이 순수 학습 효과로 해석되는 근거)
- `--max-hours` 초과 시 저장 후 종료, 100스텝마다 어댑터 중간 저장, 절전 차단

## 3. 실측 리소스 예산 (RTX 5060 Laptop 8GB, 스모크 실측)

| 항목 | 실측값 | 의미 |
|---|---|---|
| 학습 peak VRAM (2B bf16 LoRA) | 6.41GB | 한도 7.0GB 내. 다른 GPU 커널 전부 종료 필수 |
| 학습 속도 | ~1.7초/항목 | 증강×2 (18,082항목) 1에폭 ≈ **8.5시간** |
| 평가 속도 (어댑터 포함) | ~0.85초/샘플 | holdout 300 ≈ **4.5분** |
| 스크리닝 미니 학습 | `--max-samples 1000 --max-steps 300` | ≈ 1.5시간 (프롬프트 후보 1차 거름용) |

OOM 발생 시 조이는 순서: ① `--max-pixels` 축소(224px까지 zero-shot 성능 무손실 확인됨) → ② `--load-4bit` → ③ grad-accum 유지한 채 LoRA rank 축소.

## 4. 실행 명령 모음

```powershell
.venv\Scripts\Activate.ps1

# 스모크 (사이클 검증, ~3분)
python scripts/train.py --run-name smoke --max-samples 12 --aug-mult 2 --grad-accum 4 --max-steps 5

# 밤 배치 본학습 (기준점 exp01)
python scripts/train.py --run-name exp01_aug2_lr1e4 --aug-mult 2 --max-hours 10

# 평가 (학습 종료 메시지에도 같은 명령이 출력됨)
python scripts/eval_zero_shot.py --model ./models/Qwen3-VL-2B-Instruct --adapter ./outputs/runs/exp01_aug2_lr1e4/adapter

# 4B 스케일업 (4bit 필수)
python scripts/train.py --model ./models/Qwen3-VL-4B-Instruct --load-4bit --run-name exp0N_4b_best레시피 --max-hours 10
python scripts/eval_zero_shot.py --model ./models/Qwen3-VL-4B-Instruct --adapter ./outputs/runs/exp0N_4b_best레시피/adapter --load-4bit
```

**run-name 컨벤션**: `exp<번호>_<바꾼 변수>` (예: `exp02_aug4`, `exp03_lr5e5`, `exp05_v2temporal`). 전체 설정은 `run_config.json`에 자동 저장되므로 이름에 다 넣을 필요 없음.

**프롬프트를 바꾼 실험**은 학습·평가 모두 `--prompt <이름>`을 붙인다 (train.py 종료 메시지가 올바른 평가 명령을 출력해줌). 노트북 레지스트리를 쓰면 자동으로 세트가 맞춰진다.

## 5. 실험 로드맵 (우선순위순 — 변수 하나씩)

### 5.1 1단계: 기준점 (exp01) — 최우선
- `exp01_aug2_lr1e4`: 증강×2, lr 1e-4, LoRA r16/α32, 1에폭, 현행 프롬프트
- **이 결과가 나오기 전에는 다른 실험을 걸지 않는다.** 모든 후속 실험의 비교 대상.

**결과 해석 분기**:
| exp01 acc_shuffled | 해석 | 다음 행동 |
|---|---|---|
| > 20% | 레시피 방향 유효 | 5.2로 진행 (변수 최적화) |
| 8~20% | 학습은 되는데 부족 | 에폭 2로 연장 + 증강×4 우선 |
| ≤ 8% (무작위+노이즈) | 학습 신호 자체가 약함 | loss 곡선 확인 → lr 스윕 먼저, 그래도 안 되면 출력 형식(5.3)을 앞당김 |

### 5.2 2단계: 학습 하이퍼파라미터 (밤 배치 1회 = 실험 1개)
1. **증강 배수**: ×2 vs ×4 — ×4는 1에폭 17시간이므로 `--max-hours 10`으로 자르거나 base 샘플 절반 사용
2. **학습률**: 5e-5 / 1e-4(기준) / 2e-4
3. **LoRA rank**: 16(기준) vs 32 (`--lora-r 32 --lora-alpha 64`)
4. **에폭**: 1 vs 2 (2에폭은 증강×2 기준 ~17시간 → 주말 배치)

### 5.3 3단계: 출력 형식 · 프롬프트 (학습·평가 세트 교체 필수)

**프롬프트 실험 절차**: `scripts/prompts.py`의 PROMPTS에 후보 추가 → 학습·평가 양쪽에 `--prompt <이름>` (노트북 레지스트리에 `prompt="이름"` 한 줄이면 자동 세트 적용). 어느 프롬프트로 잰 결과인지 experiments.csv `prompt` 컬럼에 남는다.

1. **입력 프롬프트 변형**: 시간 키워드 강조(`v2_temporal` 초안 등록됨 — 텍스트 EDA 산출물로 개선), 캡션을 콤마/세미콜론/`then` 기준으로 하위 문장 분할 후 프레임별로 매칭 제시(§6.1 연계)
2. **24-순열 분류형 출력**: 답을 `"2413"` 같은 4자리 토큰열로 학습 — 순열 전체에 손실이 고르게 걸림. 프롬프트만으로는 불가: train.py의 `target_text` 생성부와 eval의 파서를 같이 수정
3. **제약 디코딩**: 생성 시 유효 순열만 허용 (추론 전용, 학습 불필요 — 어느 레시피에나 덧붙일 수 있는 보너스)
4. **CoT**: 학습 데이터에 CoT 답변이 없으므로 합성이 필요 — 비용 대비 후순위. 규정상 외부 API 전처리 3만 원 한도 내 합성은 가능
- 후보가 많으면 **미니 학습(1.5시간)으로 1차 스크리닝** 후 승자만 밤 배치

### 5.4 4단계: 해상도 재검증 (이미지 EDA 로드맵 3.4절 연계)
- 최고 레시피 고정 → `--max-pixels`만 224/336/448급으로 바꿔 학습·평가
- zero-shot에서는 차이가 없었지만 파인튜닝 후에는 병목이 화질로 이동할 수 있음 (EDA 담당 가설 검증)

### 5.5 5단계: Qwen3-VL-4B 스케일업 (레시피 확정 후)
- 2B 최고 레시피 그대로 + `--load-4bit`
- **lr만 재탐색** (모델이 커지면 최적 lr이 보통 낮아짐: 기준 lr과 그 절반, 2회)
- 2B 대비 acc_shuffled가 +4%p 이상이면 4B 채택, 아니면 2B 유지 (추론 속도 이점)

### 5.6 6단계: 제출
- 최고 모델로 `src/inference.py` 변형해 test 추론 → 제출 (1일 2회 제한, holdout 검증 후 확인용으로만)
- 제출 전 `HF_HUB_OFFLINE=1` 오프라인 리허설 + 추론 시간 확인 (24h/819샘플 ≈ 105초/샘플 한도, 현재 ~1초로 여유)

## 6. 미착수 백업 아이디어 (`modeling_strategy.md`에서 이관, 2026-07-13)

`modeling_strategy.md`는 zero-shot 비교 이전에 쓰인 초기 브레인스토밍 문서라 §9의 실측 기반 설계 기록과 대부분 겹쳐 삭제했다. 아직 어디에도 반영되지 않은 고유 아이디어 2개만 남긴다.

1. **시간 키워드 빈도 분석**: 캡션의 `then`/`after`/`first`/`finally` 등 시간 관계 접속사 빈도를 측정해 프롬프트 설계(`v2_temporal`, §5.3)나 학습 가중치에 반영. 텍스트 EDA 담당 산출물과 연결.
2. **CLIP dual-encoder 코사인 유사도 랭킹 (서브 접근법)**: 24개 순열의 프레임 시퀀스 임베딩과 캡션 임베딩 유사도로 순서를 고르는 경량 방식. VLM 파인튜닝이 막히는 경우의 대안/속도 비교용 벤치마크로만 고려 (주력 아님).

- **텍스트 EDA →**: 오답 파일(`errors_*_<run>.csv`)을 넘기고, 프롬프트 후보·문장 변형 전처리를 받아 5.3에 투입

- **전처리 데이터가 오면**: train.py는 `train.csv` 형식(Id, Input_1~4, Sentence, Answer)만 맞으면 `--data-dir`로 바로 학습 가능

## 8. 함정 (실험 단계 추가분 — 기존 목록은 HANDOVER.md §6)

1. **학습 중 다른 GPU 커널 금지** — 노트북 커널 하나가 4GB를 물고 있으면 OOM. 학습은 항상 별도 프로세스로
2. **exp 결과를 노트북 셀 출력으로만 남기지 말 것** — experiments.csv가 유일한 장부. 셀 출력은 휘발됨
3. **프롬프트 실험에서 평가만 바꾸는 실수** — 학습 프롬프트와 다르면 성능이 떨어지는 게 정상이고, 그 숫자는 아무것도 증명하지 않음
4. **acc_shuffled ±4%p 이내를 개선이라고 부르지 말 것** — 다음 변수로 넘어가는 게 이득
5. **밤 배치 전 스모크 습관** — 코드를 조금이라도 고쳤으면 `--max-samples 12 --max-steps 5`로 3분 확인 후 걸 것
6. **노트북에서 `!`로 띄운 학습은 커널을 죽여도 고아 프로세스로 살아남을 수 있다** — 중단 시 반드시 프로세스 트리 확인 후 종료(taskkill /T), GPU 여유 확인 후 재시작 (현행 ④셀은 Popen 방식이라 PID가 출력됨)

## 9. 설계 기록 — 베이스라인 모델 설정부터 현재 파이프라인까지 (2026-07-13)

> 각 결정의 "왜"를 남긴다. 본선 보고서(해결 전략 40점, 데이터 활용 15점, 모델 설계 15점) 초안 재료.

### 9.1 1단계: 베이스라인 모델 설정

- **후보 5개 zero-shot 비교** (holdout 300, 상세: `outputs/zero_shot_report.md`): Qwen2-VL-2B / 2.5-VL-3B / 2.5-VL-7B / Qwen3-VL-2B / Qwen3-VL-4B
- **평가 조건**: RTX 5060 Laptop 8GB, fp16 또는 4-bit(NF4), max_pixels=307200, 찍기 기준선([1,2,3,4] 고정) = 16.0%

| 모델 | 전체 정확도 | 섞인 샘플 정확도 | 안 섞인 샘플 정확도 | [1,2,3,4] 응답률 | 초/샘플 | VRAM | 구동 |
|---|---|---|---|---|---|---|---|
| Qwen2-VL-2B | 13.7% | 1.6% | 77.1% | 70.7% | 0.99 | 4.8GB | fp16 |
| Qwen2.5-VL-3B | 7.0% | 4.4% | 20.8% | 13.3% | 1.24 | 3.0GB | 4bit |
| Qwen2.5-VL-7B | 11.0% | 2.4% | 56.2% | 37.3% | 1.74 | 7.0GB | 4bit |
| Qwen3-VL-2B | 14.0% | 0.8% | 83.3% | 57.3% | **0.69** | 4.7GB | fp16 |
| **Qwen3-VL-4B** | **16.3%** | 2.0% | **91.7%** | 53.0% | 1.14 | 3.8GB | 4bit |

- **핵심 발견**: 전체 정확도(7.0~16.3%)는 전부 "[1,2,3,4] 응답 빈도" 착시 — 안 섞인 샘플이 원본 train의 16%라서, identity 응답이 잦은 모델(Qwen3-4B 53%, Qwen2-2B 70.7%)일수록 전체 점수가 부풀려짐. 섞인 샘플 정확도는 전 모델 0.8~4.4% = 무작위(1/24 ≈ 4.2%)와 동급 → **zero-shot 순위로 모델 잠재력 판단 금지**, 5개 모두 출발선 동일
- **속도·자원 관찰**: Qwen3-VL-2B가 최속(0.69초/샘플, fp16으로도 4.7GB에 그침) — 레시피 반복 실험에 가장 유리. Qwen3-VL-4B는 4bit 기준 VRAM 3.8GB로 오히려 2B(fp16)보다 여유 — 세대가 최신이라 양자화 손실이 적은 것으로 추정
- **선정 논리**: 성능이 아니라 파인튜닝 적합성으로 선택 —
  - **레시피 개발 = Qwen3-VL-2B**: 최고 속도(0.69초/샘플, 평가 회전 최속) + 8GB에서 4bit 없이 fp16/bf16 학습 가능
  - **최종 후보 = Qwen3-VL-4B**: 같은 세대의 더 큰 모델이라 레시피 이식이 성립 (세대가 다르면 프롬프트 민감도·학습 특성이 달라 이식 실패 위험). 4bit 기준 VRAM 3.8GB로 오히려 여유
  - 2.5세대(3B/7B)는 속도·메모리 대비 이점 없어 탈락
- **평가 기준 확정**: holdout_300 고정 + 핵심 지표 = acc_shuffled (identity 착시 차단)

### 9.2 2단계: 데이터 설계

- **재셔플 증강** (`train.py --aug-mult`): 같은 4프레임의 제시 순서를 순열로 바꾸고 시간순 라벨을 자동 재계산 (변형 0 = 원본 유지). 목적: ① 데이터 배수 확장 ② identity 편향 제거 — 원본 train의 16%가 안 섞인 샘플이라 그대로 학습하면 [1,2,3,4] 찍기를 강화함. 라벨 변환의 정합성은 단위 검증 완료(모든 변형이 동일한 시간순 파일 열을 복원, eval 파서 역변환과 일치)
- **검증셋 오염 방지**: 학습 제외 Id = holdout_300 ∪ stratified_valid = **494개** (train.py에 하드코딩된 자동 제외). holdout_300이 공식 평가셋, stratified_valid(팀원 제작, 6개 중복)는 Fine-grained/Scene Cut 유형별 보조 분석용
- **라벨 방향 규약**: 데이터의 Answer는 제출 형식("각 이미지의 원래 위치"), 모델 출력은 시간순 이미지 번호 — 학습 타깃은 Answer의 역순열, 평가 시 파서가 다시 역변환 (베이스라인 노트북 규약 유지)

### 9.3 3단계: 학습 파이프라인 (`scripts/train.py`)

- **구성**: LoRA r=16/α=32 (LM attention q/k/v/o), bf16, gradient checkpointing, batch 1 + accumulation 16, AdamW + cosine + warmup 3%, lr 1e-4
- **손실 설계**: 프롬프트·이미지 토큰은 -100 마스킹, 정답 텍스트 토큰만 지도 (프롬프트 암기 방지)
- **학습 프롬프트 = 평가 프롬프트** 동일 문자열 → zero-shot 대비 상승분이 순수 학습 효과로 해석됨
- **8GB 대응 실측**: peak 6.41GB (한도 7.0GB 내). OOM 시: max_pixels 축소 → 4bit → rank 축소 순
- **밤샘 안전장치**: `--max-hours` 초과 시 저장 후 종료, 100 옵티마이저 스텝마다 어댑터 중간 저장, 절전 차단, VRAM 확보 대기, KeyboardInterrupt에도 저장

### 9.4 4단계: 평가 파이프라인 확장 (`eval_zero_shot.py`)

- `--adapter`로 PEFT 어댑터를 베이스에 얹어 평가 (zero-shot과 동일 로직·동일 채점)
- experiments.csv에 `acc_shuffled`/`acc_identity`/`prompt` 컬럼 추가, 어댑터 실험은 `model_id = <모델>+<run이름>`으로 구분
- 실험별 오답 상세 `errors_<모델>_<run>.csv` 자동 생성 (텍스트 EDA 공유용)

### 9.5 5단계: 프롬프트 레지스트리 (`scripts/prompts.py`)

- 프롬프트를 코드에서 분리해 이름으로 등록 (`v1_list` = 베이스라인 원본, `v2_temporal` = 시간 키워드 강조 초안)
- train/eval 양쪽이 `--prompt <이름>` 공유 → **학습·평가 프롬프트 세트가 구조적으로 강제**되고, 어떤 프롬프트로 잰 결과인지 장부에 남음
- 한계: 출력 형식 변경(24-순열 분류 등)은 target_text 생성부 + 파서 수정이 별도로 필요

### 9.6 6단계: 실험 운영 설계 (`Train_Experiments.ipynb` + `scripts/watch_train.py`)

- **노트북 = 조종석, 스크립트 = 엔진**: 학습은 항상 별도 프로세스 (커널 크래시 무관, VRAM 완전 해제, 대회 재현성 요건인 .py 유지)
- **실험 레지스트리**: 실험 1개 = 딕셔너리 한 줄(기본값과 다른 것만). 유형 1(하이퍼파라미터)과 유형 2(프롬프트)를 같은 인터페이스로 관리
- **"모두 실행" = 풀 파이프라인**: 스모크(3분) → 본학습(백그라운드 Popen, 콘솔은 `console.log`) → 진행 게이지(30초 갱신, 완료 시 자동 통과) → holdout 평가 → loss 곡선·비교표·오답 분석
- **진행 감시**: 게이지는 `train_log.csv`만 읽으므로 GPU 무부하. 노트북 밖에서는 `python scripts/watch_train.py --run <이름>` (커널이 죽어도 학습 프로세스는 생존 — 그 경우 게이지 셀부터 재실행)
