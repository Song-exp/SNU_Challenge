# 보고서 — gemma4 문장 라벨링 (7/17)

> 로컬 gemma4로 문장 구조 라벨 900개를 추출한 작업의 전 과정 기록.
> 산출물: `outputs/gemma_labels/labels.jsonl` | 추출기: `scripts/gemma_label_sentences.py`
> 배경 설계: 팀 파이프라인 문서(문장 유형화 §1-1)의 "LLM 제로샷 라벨링"을 성분 추출 형태로 축소 적용.

## 1. 목적

문장에서 순서 단서의 구조(주체·이벤트·카메라 표현·시간 표지)를 기계 라벨로 추출해 세 용도로 회수:
① **타깃 증강 축** — 약점 세그먼트 식별·가중 (exp15), ② **오답 분석 공변량**, ③ **CoT 재도전 재료** (exp12의 spacy 분해를 대체, 미니 게이트 전제).

착수 근거 (7/17 새벽 실측 2건):
- exp12(CoT SFT) 기각의 주범이 **spacy 분해 품질**로 판명 — 분해가 깨진 문장(48%)에서 -7.4%p 집중, 깨끗한 문장은 동률. 단 깨끗해도 이득은 없었음 → 품질 개선은 필요조건이지 충분조건 아님
- **카메라/촬영 표현 유무가 최강 정확도 인자** — exp07 기준 카메라O 62.5% vs X 35.6% (+27%p, 길이 통제 후 +16.6%p). test는 카메라O 43.8%뿐 → 과반이 약점 구간

## 2. 인프라 — WSL ollama (한글 경로 버그 우회)

- Windows 네이티브 ollama는 한글 사용자 경로에서 gemma4 계열 로드에 버그 → **WSL ollama, 포트 11435** 사용 (기존 7eleven 프로젝트 런북 재활용)
- 기동: `wsl -- bash -c "OLLAMA_MODELS=/mnt/c/Users/송정현/.ollama/models OLLAMA_HOST=0.0.0.0:11435 ollama serve"` (백그라운드)
- ⚠️ **`think: false` 필수** — gemma4는 복잡한 프롬프트에서 thinking 모드로 들어가는데, ollama가 그 출력을 response에서 걸러내 **빈 응답**이 됨 (num_predict만 소모). 최초 10문장 실험이 전부 이걸로 실패했음
- 8GB GPU에 12b는 부분 오프로드(GPU 65%/CPU 35%) → GPU 사용률·발열 낮은 게 정상 (CPU 병목)

## 3. 모델 선정 — 12b 채택 (e4b 기각, 26b 미실험)

같은 3문장(함정 포함)으로 비교:

| | 12b | e4b |
|---|---|---|
| 속도 (로드 제외) | 27.7초/문장 | 23.3초 (**16% 차이뿐** — 둘 다 부분 오프로드라 격차 없음) |
| camera/viewer 불리언 | 3/3 정답 (함정 통과) | 3/3 정답 (동급) |
| events/markers 재현율 | 우위 | "misses a buoy" 이벤트, "as" 표지 등 누락 |

→ 속도 이득이 없으니 구조 추출 품질로 12b 확정. **n_events를 개수로 쓰는 설계라 e4b의 과소 분할은 직접 손해.**

## 4. 추출 스키마와 프롬프트

문장당 JSON 5필드: `subjects`(주체 목록) / `events`(시간순 사건 구절) / `camera_language`(촬영·편집 언어, "cuts through water" 같은 물리 동작 제외) / `viewer_language`("we see" 류 시청자 시점) / `temporal_markers`(순서 표지 단어).
파생: `n_events`, `n_subj`, `has_marker`, `clause_type`. 프롬프트 전문은 `scripts/gemma_label_sentences.py` (역할 부여 + 규칙 4 + few-shot 2, 사용자 검수본).

정규식 대비 gemma의 개선 (10문장 사전 실험): 오탐 제거("cuts through water"→X), 미탐 발굴("we see/is shown" 시청자 시점 범주).

## 5. 실행 결과 (7/17 02:15 ~ 09:35)

- **900/900 성공, 파싱 실패 0** — holdout 300 전량 + train 미니 풀(시드 42 고정 1000개)의 앞 600
- 29.3초/문장, 총 7.3시간. train 쪽을 "시드 고정 풀의 앞 600"으로 잡아 **CoT 미니 학습셋(1000)과 정확히 겹침** — 확장 시 나머지 400만 추가 (`--train-count 1000`, ~3.3h)
- 분포: camera 40.7% / viewer 9.6% / 표지 보유 75.1% / 이벤트 수 최빈값 4
- **이어받기**: 같은 명령 재실행 시 성공한 Id는 스킵, 실패분만 재시도 (JSONL 증분 기록)

## 6. 검증 발견 (holdout 300 × exp07 예측 대조)

| 축 | O | X | 격차 | 판정 |
|---|---|---|---|---|
| gemma camera | 64.4% | 37.2% | **+27.3%p** | 정규식(+26.9)과 일치율 92.1% — 축 유효성 확정 |
| gemma viewer | 42.1% | 48.9% | **-6.8%p** (n=19) | **역방향!** 시청자 시점 문장이 오히려 어려움 → camera와 분리 유지, 병합 금지 |
| n_events | 1개 16.7% → 2개 24.2% → 3개 61.3% → 5개 74.5% | | 계단식 | **camera보다 가파른 인자** — 이벤트 1~2개(33%)가 진짜 약점 |

## 7. 활용 계획

1. **exp15 타깃 증강 (즉시)**: 가중 축은 정규식 camera 사용 (92.1% 일치로 전량 커버 가능, gemma 라벨은 검증·정밀 분석 담당) — 카메라X aug4 / 카메라O aug2
2. **CoT 재도전 (게이트 통과 시)**: gemma events를 [Story Analysis] target으로 미니 1000 SFT → exp06 미니 기준 대폭 승리 시에만 확전. 나머지 400 라벨 필요 (~3.3h)
3. **오답 분석**: n_events·multi_subject를 공변량으로 어려운 Type-2 해부
4. **본선 자산**: 파이프라인 문서 1장(유형화)의 실측 기반 — media_deixis(camera·viewer 분리), 문장 유형 정밀화의 출발점

## 8. 비용·한계

- 외부 API 0원 유지 (전부 로컬). GPU 점유 7.3h (부분 오프로드라 학습과 동시 실행 불가)
- 라벨은 gemma 판정이지 사람 검수본 아님 — 파이프라인 문서의 κ 게이트(사람 300개 검수)는 미실시, 예선 내 미계획
- viewer 발견은 n=19라 잠정적 — train 600 라벨로 재확인 가능
- 12b도 완벽하지 않음: 주어 없는 파편 이벤트가 드물게 남음 (spacy보다는 훨씬 적음)
